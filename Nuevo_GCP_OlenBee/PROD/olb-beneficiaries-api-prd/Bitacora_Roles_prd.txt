===============================


SERVICE_ACCOUNT1="olenbee-deployment-pipeline@olb-beneficiaries-api-prd-6530.iam.gserviceaccount.com"

Adding network config to the cloud run instance:

Cloud Run has what we call a service agent that needs permissions to deploy the service or job in the connector or subnet.
First you need to be sure that you use the connector we have created in your networking configuration

Inside Clñoud Run instance network configuration
|
Select:Connect to a VPC for outbound traffic
 |
 |__Selet: Use Serverless VPC Access connectors
    |
    |__Choose: olb-prd-network-glb-netspoke: Serverless VPC Access Connector “olb-prd-connector” (in host project “olb-netsp

Then, we added an IAM role "Serverless VPC Access User" in the host project "olb-netspoke-prd" for the Cloud Run service agent which has this form : service-<PROJECT_NUMBER>@serverless-robot-prod.iam.gserviceaccount.com
So in our case the project number is : 232450500396 thus our cloud run service agent is : service-232450500396@serverless-robot-prod.iam.gserviceaccount.com

First check the roles for that SA:

Example:

gcloud projects get-iam-policy olb-netspoke-prd-39fa --format="json" | jq '.bindings[] | select(.members[]? | contains("service-232450500396@serverless-robot-prod.iam.gserviceaccount.com"))'
{
  "members": [
    "serviceAccount:232450500396@cloudservices.gserviceaccount.com",
    "serviceAccount:service-232450500396@serverless-robot-prod.iam.gserviceaccount.com"
  ],
  "role": "roles/vpcaccess.user"
}

If its not assined, please proceed to assign the role in the network project (olb-netspoke-prd-39fa):

gcloud projects add-iam-policy-binding YOUR_PROJECT_ID \
    --member="serviceAccount:service-<Project Number>@serverless-robot-prod.iam.gserviceaccount.com" \
    --role="roles/editor"
